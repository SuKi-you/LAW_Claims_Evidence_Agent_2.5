
# AI-Agent-2.5 法律诉求与证据整理助手

> 面向婚姻家事场景的轻量级法律诉求识别与证据整理 Demo。  
> 本项目不提供正式法律意见，而是帮助用户把口语化案情转化为可确认的法律诉求，并在用户确认诉求后生成克制、可执行的证据准备清单。

---

## 一、项目背景

在婚姻家事纠纷中，普通用户经常会用非常口语化的方式描述问题，例如：

```text
我想离婚。
他偷偷把钱转走了。
孩子一直是我带，我想争取孩子。
他打过我，还威胁我。
他给小三转了很多钱。
```

这些表达背后可能对应不同法律诉求，例如离婚、子女抚养权、抚养费、探望权、财产分割、财产转移、追回第三者赠与、家暴 / 人身安全保护、离婚损害赔偿等。

但是，如果直接让大模型根据用户案情输出诉求和证据，容易出现两个问题：

### 1. 诉求识别容易过度召回

用户只说"我想离婚"，系统可能自动扩展出子女抚养、财产分割、家暴保护、损害赔偿等未确认诉求。

用户说"没有孩子"，系统如果没有稳定识别否定事实，仍可能生成子女抚养权、抚养费、探望权等诉求。

### 2. 证据清单容易泛化或越界

证据阶段如果直接根据原始案情生成证据，系统可能把用户没有确认的事实也转化为证据任务。

例如：

```text
用户只确认"离婚"
但系统输出"第三者转账记录""家暴报警记录""子女出生证明"
```

这会带来两个问题：

1. 用户体验上显得过度、混乱；
2. 产品逻辑上不符合"用户确认诉求后再生成证据"的边界。

因此，本项目设计了 Agent 2.5：

```text
Dify / LLM 负责候选生成
用户确认诉求 confirmed_claims 负责锁定证据范围
Evidence 阶段只围绕 confirmed_claims 输出证据清单
```

项目目标不是做一个完整替代律师的系统，而是做一个稳定、可演示、可测试的法律诉求与证据整理助手。

---

## 二、Agent 2.5 的逻辑流程与系统架构

### 2.1 核心逻辑流程

Agent 2.5 的主流程如下：

```text
用户输入口语案情
    ↓
Intent Discovery 识别候选诉求
    ↓
前端展示候选诉求卡片
    ↓
用户勾选 / 确认诉求 confirmed_claims
    ↓
Evidence Workflow 根据 confirmed_claims 生成证据清单
    ↓
前端展示核心证据、辅助证据、风险提示
```

这里最重要的设计是：

```text
confirmed_claims 是证据阶段的唯一诉求入口。
```

也就是说，原始案情只提供背景，真正决定证据清单范围的是用户确认后的诉求。

### 2.2 2.5 的四层架构

```text
┌──────────────────────────────┐
│  前端交互层                    │
│  - 用户输入                    │
│  - 候选诉求卡片                │
│  - 用户确认诉求                │
│  - 证据清单展示                │
│  - loading / error / fallback │
└───────────────┬──────────────┘
                ↓
┌──────────────────────────────┐
│  Dify 候选生成层               │
│  - Intent Discovery App        │
│  - Evidence App                │
│  - LLM 负责候选生成            │
└───────────────┬──────────────┘
                ↓
┌──────────────────────────────┐
│  Guard 思想控制层              │
│  - confirmed_claims 控制范围   │
│  - 否定事实排除                │
│  - 未确认诉求不得进入证据阶段   │
│  - Evidence 输出结构约束       │
└───────────────┬──────────────┘
                ↓
┌──────────────────────────────┐
│  测试与评估层                  │
│  - Golden Cases                │
│  - baseline / optimized 对比   │
│  - raw_results                 │
│  - summary / report / Excel    │
└──────────────────────────────┘
```

### 2.3 为什么是 2.5，而不是 2.0 或 3.0

本项目在设计上经历了 2.0、3.0、2.5 三个阶段的思考。

```text
2.0：工程稳定，但法律识别偏粗
3.0：法律推理更完整，但工程链路过重
2.5：保留 2.0 的轻量前端，吸收 3.0 的法律判断思想，压缩成可控的 Guard 架构
```

2.5 的核心定位是：

```text
不是完整模拟律师推理链，
而是把法律判断中的关键边界压缩成稳定、可测试、可演示的产品逻辑。
```

## 三、Dify 工作流设计

Agent 2.5 使用两个 Dify App：

```text
Intent Discovery App：识别候选诉求
Evidence App：根据 confirmed_claims 生成证据清单
```

### 3.1 Intent Discovery App

Intent Discovery App 负责从用户原始案情中识别候选诉求。

输入：

```text
用户原始案情 query
```

输出示例：

```json
{
  "candidate_claims": [
    {
      "claim_name": "离婚",
      "confidence": "high",
      "reason": "用户明确表达离婚意愿"
    }
  ],
  "excluded_claims": [
    {
      "claim_name": "子女抚养权",
      "reason": "用户明确表示没有孩子"
    }
  ],
  "missing_info": []
}
```

当前核心标准诉求包括：

```text
离婚
子女抚养权
抚养费
探望权
财产分割
家暴 / 人身安全保护
离婚损害赔偿
追回第三者赠与
财产转移
```

Intent 阶段重点解决的问题：

1. 标准化诉求名称；
2. 识别用户明确表达的诉求；
3. 根据否定事实排除不应出现的诉求；
4. 避免因为"离婚"自动扩展全部附带诉求；
5. 区分"普通财产分割""财产转移""追回第三者赠与"等相近但不同的诉求。

---

### 3.2 Evidence App

Evidence App 负责根据 confirmed_claims 生成证据清单。

输入：

```text
用户原始案情 query
用户确认诉求 confirmed_claims
```

工作流结构：

```text
Start - 用户输入与确认诉求
    ↓
Query_Cleaner
    ↓
Knowledge Retrieval
    ↓
Evidence Light Generator
    ↓
Output - final_evidence_list_for_user
```

各节点作用：

### Start - 用户输入与确认诉求

接收用户原始输入和 confirmed_claims。  
confirmed_claims 是 Evidence 阶段的范围入口。

### Query_Cleaner

将用户口语输入和 confirmed_claims 转化为适合知识库检索的关键词。

关键原则：

```text
只围绕 confirmed_claims 中已确认的诉求生成检索词。
不得根据原始案情扩展未确认诉求。
```

### Knowledge Retrieval

从婚姻家事知识库中召回相关规则、证据方向和风险提示。

知识库内容包括：

```text
离婚基础规则
子女抚养规则
家暴与人身安全保护规则
财产分割规则
第三者赠与与财产转移规则
通用民事证据规则
```

### Evidence Light Generator

根据 confirmed_claims 和知识库召回内容输出 C 端证据清单。

输出结构：

```json
{
  "final_evidence_list_for_user": {
    "core_evidence": [
      {
        "evidence_name": "结婚证或婚姻登记信息",
        "priority": "核心证据",
        "note": "用于证明婚姻关系存续。",
        "prepared": false
      }
    ],
    "auxiliary_evidence": []
  },
  "risk_tips": [
    "诉讼离婚需围绕感情破裂准备证据。"
  ]
}
```

Evidence 阶段禁止输出：

```text
proof_path
legal_elements
facts_to_prove
matched_rules
factors
claim_cards
evidence_candidates
```

这些内容属于系统内部推理变量，不直接展示给 C 端用户。

## 四、用户流程

### 4.1 用户输入案情

用户输入自然语言案情，例如：

```text
我想离婚，婚后买了一套房和一辆车，现在想分割夫妻共同财产。
```

### 4.2 系统识别候选诉求

Intent Discovery App 输出候选诉求：

```text
离婚
财产分割
```

### 4.3 用户确认诉求

前端以卡片形式展示候选诉求，由用户确认。

例如用户最终确认：

```json
["财产分割"]
```

### 4.4 系统生成证据清单

Evidence App 只围绕"财产分割"输出证据清单，例如：

```text
核心证据：
- 房产证或不动产登记信息
- 购房合同或出资凭证
- 车辆登记证书
- 夫妻共同财产清单
- 银行流水或银行存款证明
```

不会因为原始案情中提到"离婚"，就额外输出离婚诉讼材料；也不会输出子女、家暴、第三者相关证据。

### 4.5 用户查看并准备证据

前端展示：

```text
证据名称
核心 / 辅助
准备提示
是否已准备
风险提示
```

C 端用户看到的是极简清单，而不是复杂法律推理过程。

## 五、3.0 律师思考模型与 2.5 的吸收压缩

### 5.1 3.0 的律师思考模型

3.0 阶段曾尝试构建更完整的"律师式推理链"。

其核心过程可以概括为：

```text
用户口语输入
    ↓
按句切分
    ↓
原子事实因子抽取
    ↓
法律化事实因子
    ↓
主事实因子 / 加权因子 / 降权因子 / 排除因子
    ↓
法条 / 裁判规则匹配
    ↓
候选诉求生成
    ↓
用户确认诉求
    ↓
证明路径
    ↓
待证事实
    ↓
证据线索
    ↓
证据收敛
    ↓
C 端证据清单
```

这个模型的价值在于：它试图解决法律 Agent 不能只靠关键词匹配的问题。

例如：

```text
"没有孩子"不是普通事实，而是排除子女抚养权、抚养费、探望权的排除因子。

"他偷偷把钱转给亲戚"不是普通财产事实，而是更应触发财产转移。

"他给小三转钱"不仅是财产问题，还可能触发追回第三者赠与。

"推了一下但没受伤、没报警"不应直接生成强家暴结论。
```

### 5.2 事实因子思想

3.0 中的事实因子并不是简单关键词，而是用来判断法律触发强度的内部变量。

可以分为：

```text
主事实因子：直接触发某类诉求
加权因子：增强某类诉求的置信度
降权因子：降低某类诉求的置信度
排除因子：阻止某类诉求生成
事实细节：作为背景信息，不直接触发诉求
```

例如：

```text
"我想离婚" → 离婚诉求的主事实因子
"没有孩子" → 子女类诉求的排除因子
"一直是我照顾孩子" → 子女抚养权的加权因子
"只是推了一下，没有受伤也没报警" → 家暴保护或损害赔偿的降权因子
"转到亲戚名下" → 财产转移的加权因子
```

### 5.3 3.0 为什么没有直接落地

3.0 的问题不在于思想错误，而在于当前 Demo 阶段工程负担过重。

主要问题包括：

1. 中间变量太多，链路过长；
2. 类型结构复杂，调试成本高；
3. 用户看不到大量内部推理层，展示价值有限；
4. LLM 需要稳定输出大量结构化字段，失败风险高；
5. 证据阶段容易出现证据爆炸；
6. 不利于快速自动化 eval；
7. 容易让 coding agent 在实现时过度发挥，改坏主状态机。

因此，2.5 没有直接照搬 3.0 的完整链路，而是吸收其中最有价值的思想。

---

### 5.4 2.5 如何吸收 3.0 的事实因子思想

2.5 没有显式实现完整 facts pipeline，但把事实因子的核心思想压缩进了诉求识别规则和证据范围控制中。

#### 1. 主事实因子 → 候选诉求触发规则

例如：

```text
"我想离婚" → 触发"离婚"
"我要争取孩子" → 触发"子女抚养权"
"要求对方支付抚养费" → 触发"抚养费"
"把钱转走 / 转到亲戚名下" → 触发"财产转移"
"给小三转账" → 触发"追回第三者赠与"
```

#### 2. 排除因子 → excluded_claims

例如：

```text
"没有孩子" → 排除子女抚养权、抚养费、探望权
"没有共同财产争议" → 排除财产分割
"没有家暴" → 排除家暴 / 人身安全保护
```

这些排除规则让系统不会只因为用户提到"离婚"就自动生成所有相关诉求。

#### 3. 加权 / 降权思想 → confidence 与候选排序

2.5 没有暴露复杂权重分数，但保留了加权、降权思想。

例如：

```text
"孩子一直是我照顾" → 增强子女抚养权置信度
"对方长期不让看孩子" → 增强探望权置信度
"大额转账、频繁转账、转给亲戚" → 增强财产转移置信度
"只是轻微推搡，没有受伤、没有报警" → 降低强家暴结论
```

在产品上，这些思想体现为：

```text
candidate_claims 的 confidence
excluded_claims 的 reason
候选诉求展示顺序
不确定信息进入 missing_info
```

#### 4. 证明路径 / 待证事实 → 压缩为证据模板与知识库 chunk

3.0 原本会把诉求拆成证明路径和待证事实。  
2.5 没有把这些内部层直接展示出来，而是把它们压缩到知识库 chunk 和 Evidence Prompt 中。

例如，财产转移对应的知识库 chunk 会围绕：

```text
异常转账
账户流水
共同财产属性
转移时间线
财产保全
少分或不分风险
```

这样既保留了律师思考中的"证明什么、用什么证明"的核心逻辑，又避免在前端展示复杂推理链。

---

### 5.5 chunk 思想如何融入 2.5

2.5 的知识库不是简单堆法条，而是按业务场景拆成更适合召回的 chunk。

chunk 设计思路：

```text
一个 chunk 对应一个具体法律场景或证据任务
每个 chunk 尽量包含：
- 适用诉求
- 触发事实
- 证据方向
- 风险提示
- 排除边界
```

例如：

```text
财产转移 chunk：
适用诉求：财产转移
触发事实：隐藏、转移、变卖、毁损、挥霍夫妻共同财产
证据方向：银行流水、异常转账、财产线索、共同财产凭证
风险提示：及时固定证据、必要时考虑财产保全、分割时可能少分或不分
排除边界：不等同于第三者赠与，除非 confirmed_claims 同时包含追回第三者赠与
```

chunk 的作用是：

1. 帮助 Query Cleaner 更精准召回；
2. 降低 Evidence LLM 直接幻想证据的概率；
3. 让证据清单更贴近具体诉求；
4. 把 3.0 的证明路径思想压缩成可维护的知识库单元。

---

### 5.6 2.0 轻量化前端如何融入 2.5

2.0 的优势是工程稳定：

```text
前端状态较少
用户路径清晰
页面结构简单
交互成本低
适合 Demo 展示
```

2.5 没有推翻 2.0 前端，而是保留其轻量化特点：

```text
输入框
候选诉求卡片
用户确认按钮
证据清单展示
```

同时在逻辑层增强：

```text
2.0 前端稳定壳
    +
2.5 confirmed_claims 范围控制
    +
Dify 双工作流
    +
RAG 知识库 chunk
    +
Golden Eval 测试
```

这样做的好处是：

1. 不增加复杂状态机；
2. 不把律师后台逻辑暴露给 C 端；
3. 用户只看到"诉求确认"和"证据准备"；
4. 项目仍然适合快速演示和作品集展示。

## 六、如何进行优化与评测

### 6.1 优化目标

本项目的优化目标不是追求一次性满分，而是验证 Agent 2.5 的关键设计是否有效：

```text
1. 诉求识别是否更准确
2. 否定事实是否能排除不应出现的诉求
3. Evidence 阶段是否严格围绕 confirmed_claims
4. JSON 输出结构是否稳定
5. 风险提示是否稳定
6. 财产转移、第三者赠与、家暴保护等复杂场景是否得到改善
```

### 6.2 优化方法

主要优化动作包括：

#### 1. 标准化诉求名称

统一使用固定诉求名，避免同一诉求出现多个表达。

例如：

```text
"家暴/人身安全保护"
统一为：
"家暴 / 人身安全保护"
```

#### 2. 强化 Intent 触发与排除规则

例如：

```text
"没有孩子"必须排除子女抚养权、抚养费、探望权。
"孩子跟对方生活，但我想每周看孩子"应触发探望权，而不是抚养权。
"偷偷把钱转到亲戚名下"应触发财产转移。
"给小三转账"应触发追回第三者赠与。
```

#### 3. 引入 confirmed_claims 范围控制

Evidence 阶段不再根据原始案情自由扩展证据，而是只根据用户确认诉求生成证据。

```text
confirmed_claims = ["离婚"]
→ 只输出离婚证据

confirmed_claims = ["财产转移"]
→ 只输出财产转移证据

confirmed_claims = ["追回第三者赠与"]
→ 只输出第三者赠与相关证据
```

#### 4. 优化 Query Cleaner

Query Cleaner 只围绕 confirmed_claims 生成检索关键词，避免把未确认事实带入知识库召回。

#### 5. 优化 Evidence 输出格式

通过 structured output 约束 Evidence App 输出固定 JSON 结构：

```text
final_evidence_list_for_user
risk_tips
```

#### 6. 针对财产转移补强最低证据方向

财产转移场景至少要覆盖：

```text
银行账户流水
异常转账记录
夫妻共同财产凭证
```

### 6.5 具体改进

#### 1. 诉求识别能力提升明显

Intent Pass Rate 从 70.00% 提升到 86.67%。

说明优化后的诉求标准化、触发规则、否定事实排除规则有效提升了系统对口语案情的识别能力。

#### 2. 证据范围控制保持稳定

Evidence Scope Pass Rate 保持 100%。

说明 confirmed_claims 作为证据阶段唯一入口的设计是有效的，系统没有把用户未确认的诉求证据混入最终清单。

#### 3. JSON 输出结构保持稳定

JSON Schema Pass Rate 保持 100%。

说明 Evidence App 的 structured output 和 Answer 输出结构稳定，前端可以较稳定地解析结果。

#### 4. 风险提示略有提升

Risk Tips Pass Rate 从 80.00% 提升到 83.33%。

说明 risk_tips required 约束和诉求相关风险提示优化起到一定作用。

#### 5. 财产转移场景改善

财产转移类 case 得到重点修复，能够更稳定输出：

```text
银行流水
异常转账
夫妻共同财产凭证
财产转移线索
财产保全风险提示
```

### 6.6 当前不足与后续方向

当前版本仍不是满分版本。

主要不足：

1. 部分 evidence_keywords 仍可能缺失；
2. 部分复杂诉求仍可能出现空证据；
3. 彩礼返还、亲子关系确认/否认等扩展诉求尚未纳入核心 9 类；
4. Dify Prompt 继续堆叠存在边界，过长 Prompt 可能降低模型稳定性。

后续更合理的优化方向不是继续无限扩写 Dify Prompt，而是：

```text
Dify 负责候选生成
本地 Evidence Guard 负责兜底、过滤和最终收敛
```

也就是说：

```text
如果 Evidence 输出为空 → 本地按 confirmed_claims 补最低证据模板
如果 risk_tips 为空 → 本地按 confirmed_claims 补风险提示
如果输出未确认诉求证据 → 本地删除
```

这样可以继续保持 2.5 的核心路线：

```text
轻量前端 + Dify 候选生成 + confirmed_claims 范围控制 + 本地 Guard 稳定收敛
```
